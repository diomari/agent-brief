# Package Improvements: Lean Project Context

## Goal

Make `agent-brief` generate a smaller, sharper `PROJECT_CONTEXT.md` that helps agents get oriented quickly without flooding the context window.

## Product direction

`/brief` should produce a compact project briefing, not a full repository audit. The output should capture only stable, high-signal information that future agents need before editing.

Prioritize:

- concise stack detection
- runnable commands
- important folders and files
- risky areas
- project conventions
- safe editing rules

Avoid:

- long source excerpts
- dependency dumps
- full directory trees
- generated files
- repeated obvious metadata
- speculative architecture claims

## Target output size

Default `PROJECT_CONTEXT.md` should be compact and stay around 50–80 lines.

`/brief --full` may generate a more detailed brief around 150–200 lines for users who explicitly want broader repository context.

Default compact section budgets:

- Stack: 5–6 lines
- Commands: 5–7 lines
- Map: 6–8 lines
- Key Files: 6–12 lines
- Risks: 4–8 lines
- Rules: 5–7 lines
- Fresh Session Prompt: 2–3 lines

If the project is large, summarize more aggressively instead of expanding the default file.

## Improvement roadmap

### 1. Make compact the default and add `--full`

Support two clear output modes:

```txt
/brief
/brief --full
```

Suggested behavior:

- `/brief`: compact default, 50–80 lines, optimized for agent working memory
- `/brief --full`: detailed mode, 150–200 lines, for broader repository briefing

Do not require a `--compact` flag for normal use. Compact should be the default product behavior.

Why this matters:

- reduces context bloat
- speeds up agent orientation
- improves usefulness after session compaction
- lowers the chance of stale or irrelevant details steering the model
- encourages agents to inspect real task-relevant files before editing
- keeps the generated context safer and easier to review

### 2. Score signals before writing

Add a signal scoring layer so only useful entries are included.

High-value signals:

- detected framework/runtime/package manager
- scripts users can run
- auth/billing/database/deployment risk markers
- top-level folders that affect implementation
- config files tied to build, deploy, or database behavior

Low-value signals:

- generic folders with no known role
- lockfiles beyond package manager detection
- duplicate framework hints
- dependencies not tied to architecture or risk

### 3. Collapse unknowns

Avoid repeating `Not detected` many times. Prefer grouped unknowns:

```md
## Unknowns

- No test command detected.
- No database/ORM signal detected.
- Deployment target unclear.
```

This keeps the main sections cleaner.

### 4. Keep architecture factual

Only include architecture claims backed by file/folder/config signals.

Good:

```md
- Routes/pages: `app/`
- Components: `components/`
- Data access: `db/`, Prisma detected
```

Avoid:

```md
- Business logic: service-oriented architecture
```

unless the repo clearly proves it.

### 5. Add project-size awareness

Detect rough project size without scanning ignored folders.

Suggested categories:

- tiny: fewer than 20 tracked source/config files
- small: 20–100 files
- medium: 100–500 files
- large: 500+ files

For medium/large projects, reduce file listings and include only top-level patterns.

### 6. Generate a machine-readable cache later

Optional future output:

```txt
.pi/brief.json
```

Use it for stable detection data, not for agent context. This lets repeated `/brief` runs compare changes without bloating `PROJECT_CONTEXT.md`.

### 7. Improve task-specific mode

For:

```txt
/brief --task "add login page"
```

Append only a short task section:

```md
## Task Lens

- Likely areas: `app/`, `components/`, auth config
- Risks: auth/session behavior
- First step: inspect existing route and form patterns before editing
```

Do not expand the whole brief.

### 8. Keep repeated runs idempotent

When `PROJECT_CONTEXT.md` exists, update it in place instead of creating another brief file. The prompt should say:

- existing context was updated
- agent should read the updated file
- user can re-run `/brief` whenever architecture context gets stale

## Proposed compact template

```md
# Project Context

Generated by `/brief`.

## Stack

- Language:
- Runtime:
- Package manager:
- Framework:
- Database / ORM:
- Deployment:

## Commands

- Install:
- Dev:
- Build:
- Typecheck:
- Lint:
- Test:
- DB/migrations:

## Map

- Entry:
- Routes/pages:
- API/server:
- UI/components:
- Logic:
- Data:
- Config/deploy:

## Key Files

- `package.json` — scripts and dependencies
- `README.md` — project docs

## Risks

- Auth:
- Billing:
- Database:
- Secrets:
- Deployment:

## Rules

- Read this file first.
- Inspect task-relevant files before editing.
- Keep diffs minimal.
- Preserve local patterns.
- Do not touch secrets, migrations, auth, billing, or deployment without approval.
- Run available checks when possible.
```

## Acceptance criteria

A future implementation should be considered successful when:

- `/brief` defaults to compact output
- default brief is around 50–80 lines
- `/brief --full` produces a more detailed 150–200 line brief
- no source files are embedded
- no secrets or `.env` files are read
- repeated `Not detected` lines are minimized
- output remains useful after context compaction
- task mode adds focus without large expansion
